What is OctoPrint?

Monitoring & remote control solution for consumer 3d printers

Architecture

Target audience & platform

  • Audience: Owners of 3d printers = makers & tinkerers
    • end users
    • rarely developers themselves
  • Platform: Small single PCB linux computers (Raspberry Pi)
    • e.g. Raspberry Pi
    • but: platform agnostic!

Challenge #1

Initial installation and keeping things up to date

Initial installation

  • Initial installation happens by end user
    • Bootstrap Python & pip install octoprint? Doesn't work for audience.
    • üí° Preconfigured Raspberry Pi image: download, flash, boot, wizard, done
  • And on something that is not a Pi?
    • Can be basically anything: huge variety in encountered environments
    • Responsibility of the user
    • Code needs to anticipate that
      • E.g. IPv6 dual stack on Windows

Keeping things up to date

  • pip install -U octoprint? Doesn't work for audience.
    • üí° Built-in update mechanism for OctoPrint & plugins
  • Rest of the environment?
    • Done by some users, not by others
      • Often prompted by scary "new version" messages during plugin installs through pip
      • Lots of conflicting information spread across the internet
    • Result: unknown environment, sometimes broken environment, hard to test, additional support overhead

What about the image itself?

  • Real challenge ("Never touch a running system")
  • Software stack quickly outdates
  • Some ancient python stacks out there
  • üí° Disable OctoPrint updates on ancient setups & provide migration path

Lessons Learned

Try to control runtime environment but stay flexible, make updates easy and use features as motivator for full image updates.

Challenge #2

Maintaining backwards compatibility for 3rd party plugins

Maintaining backwards compatibility for 3rd party plugins

  • Plugin system since v1.2.0/2015, over 170 plugins
  • Established plugin ecosystem, requires backwards compatibility to flourish
    • Disgruntled authors = no ecosystem = disgruntled users
  • Problem: Dependency updates with breaking changes
  • Problem: Code refactoring
  • Problem: Python 2 vs 3

Lessons Learned

Read changelogs, be prepared to bend over backwards to work around breaking changes, make life easy for plugin authors.

Challenge #3

Support overhead

Support overhead

  • Early days: Huge overlap between printer owners & DIY enthusiasts
    • Tinker happy
    • Happy to google around for a solution
    • Capable of fixing stuff themselves
  • These days: Lots and lots of consumers
    • Printers are marketed as plug-n-play but aren't
    • Many owners never had to tinker before
    • OctoPrint might be their first interaction w/ Open Source

Support overhead

  • Lots of development/OSS inexperienced users
    • Constant need to ask for logs & more info that "it doesn't work"
    • Tricky to explain some things, e.g. why proprietary printer won't work
      and can't "quickly be added"
  • Sometimes expectation to get support on par with huge multinational company
  • Sometimes hostile reaction when told inconvenient truths
  • Enormous overhead caused by this

Lessons Learned

Community building to delegate support, here through a forum as central place to get support - let people help each other!
